Overview

Move from a single-face native solver to a shell-aware solver with explicit face-boundary orientation transfer.

The current fishnet workbench now has direct surface sampling and a quad/net output path, but shell-wide use still needs a native adapter that preserves warp/weft orientation across adjacent faces. This plan turns the shell orientation spec into an implementation sequence that keeps the public CompositeShell API stable while moving the geometry work into C++.

Rule: the Python fallback remains only until the native extension is built. Production code should prefer the C++ path as soon as it is available.

Goals and scope

In scope

  • native face solver that accepts FreeCAD Part::Face / shell geometry directly
  • shell adapter that walks face adjacency and propagates orientation
  • explicit handling for smooth boundaries, seams, and open edges
  • compatibility with CompositeShell, TexturePlan, and grid rendering
  • Python fallback API for source checkouts before the native extension exists

Out of scope for this phase

  • manufacturing-grade multi-piece shell decomposition
  • full UI redesign
  • exact numerical parity with the prototype draper
  • removing all Python from the workbench
  • changing the public CompositeShell document object

Architecture

Native face solver

A C++ solver owns direct surface queries, sampling, projection, and pin-jointed net relaxation for one face at a time.

  • sample the face trim domain
  • seed the warp/weft lattice
  • solve and project in the face tangent frame
  • emit nodes, quads, boundary loops, and face frame metadata

Native shell adapter

A thin C++ layer above the face solver traverses adjacency, propagates orientation, and merges per-face output into a shell-level result.

  • choose a stable seed face
  • transfer local frames across adjacent faces
  • flag seams and discontinuities explicitly
  • stitch metadata for downstream consumers

Python compatibility layer

Python remains the bridge into FreeCAD document objects, the task panel, and the existing renderer APIs.

  • keep the CompositeShell API stable
  • mirror the native result structure until build availability
  • do not use the old prototype draper in production

Phase 1 — native face solver API

Objective: make the native solver operate directly on FreeCAD face geometry.

  1. Add a C++ entrypoint that accepts a single Part::Face or face-like geometry handle.
  2. Move surface sampling, projection, and frame extraction into native code.
  3. Emit a result that includes fabric_points, fabric_quads, boundary_loops, face_frames, and error/status fields.
  4. Keep the Python fallback API mirrored until the native extension is built.

The face solver should remain small and testable. It should not know about the FreeCAD GUI or document-object lifecycle.

Phase 2 — shell adapter and orientation transfer

Objective: propagate warp/weft continuity across face boundaries in native code.

  1. Build a face adjacency graph from the shell geometry.
  2. Choose a stable starting face and initial frame.
  3. For each adjacent face, transfer the previous face’s frame into the new face tangent plane.
  4. Re-orthonormalize the local frame and choose the orientation that best preserves continuity.
  5. Mark seams, hard breaks, and reseeds explicitly in the result.

Orientation policy

  • prefer continuity across smooth boundaries
  • break continuity at seams, UV flips, or highly non-smooth edges
  • do not silently guess when projection degenerates
  • keep break markers in the output so tests and previews can inspect them

Phase 3 — workbench integration

Objective: connect the native shell result to the existing workbench API.

CompositeShell

  • switch the production path to the native shell solver
  • preserve get_tex_coords(), get_boundaries(), get_lcs(), and get_strains()
  • keep DrapeStatus / DrapeError explicit

Texture renderer

  • keep TexturePlan working from get_boundaries()
  • keep MeshGridShader consuming get_tex_coords()
  • do not expose solver internals to renderer code

Plots and previews

  • keep solid-case plots showing the unwrapped net panel
  • use the same color split between source geometry and net output
  • show shell boundary breaks only when relevant

Phase 4 — fallback removal and cleanup

Objective: retire Python surface-query fallback from the production path once the native extension is built.

  • keep the Python fallback available only for source checkout / prebuild use
  • remove any production dependency on Python surface queries once C++ is in place
  • prune obsolete mesh-sampling helpers only after native coverage is stable
  • update docs and handover notes to reflect the native shell adapter

Testing

Native face tests

  • planar face solves
  • cylindrical face solves
  • concave trimmed face solves
  • boundary loops and quad extraction are correct

Native shell tests

  • two smooth adjacent faces preserve orientation
  • hard-edge transitions are marked as breaks
  • seams / discontinuities reseed cleanly
  • flipped parameterization cases do not silently twist the net

FreeCAD integration tests

  • CompositeShell still recomputes in a real document
  • TexturePlan still consumes shell boundaries
  • solid cases still show the unwrapped net panel
  • failure states remain explicit in DrapeStatus / DrapeError

Risks

  • Orientation drift: add explicit frame transfer and continuity checks at each face boundary.
  • Degenerate edges: mark them as breaks instead of guessing a frame.
  • Native API churn: keep the result structure compact and versionable.
  • Compatibility regressions: protect CompositeShell / TexturePlan integration with tests.
  • Fallback confusion: clearly separate the temporary Python fallback from the production code path.

Deliverables

  • native face solver entrypoint accepting FreeCAD face geometry
  • native shell adapter with orientation-transfer rules
  • Python fallback mirror kept only for prebuild/source use
  • updated CompositeShell integration
  • compatibility with TexturePlan and grid rendering
  • new face- and shell-oriented tests

Success criteria

  • the solver works directly on FreeCAD surface geometry in native code
  • shell face orientation stays coherent across smooth boundaries
  • seams and discontinuities are explicit and testable
  • CompositeShell keeps the same public API for downstream tools
  • TexturePlan and grid rendering continue to work from the consumer side
  • Python fallback remains only as a temporary build-time compatibility layer

Next action

Implement the native face solver first, then add the shell adapter and orientation propagation, then re-run the workbench integration tests to confirm the existing consumers still behave the same.
