Overview

Plan to remove shape2Mesh() / MeshPart.meshFromShape() from the production drape path.

The current fishnet backend still starts from a triangulated mesh. The next redesign should instead sample the real CAD surface directly, construct a pin-jointed fabric net, and solve it with a GIB-style equilibrium loop. The long-term reference is the Net algorithm used in KinDrape.

Principle: use the surface itself as the source of truth; triangulation may remain as a diagnostic or legacy comparison tool, but not as the production input pipeline.

Goal and scope

In scope

  • Direct surface queries on Part.Face / trimmed surfaces
  • Pin-jointed net data model (nodes, bars, quads)
  • GIB-style relaxation and projection to the surface
  • Boundary anchoring and seed lattice generation
  • Native solver implementation and Python adapter
  • Plot/test updates to visualize net topology rather than triangle faces

Out of scope for stage 1

  • Full UI redesign
  • Multi-piece shell stitching
  • Manufacturing allowances, flanges, or trim stock
  • Exact numerical parity with KinDrape
  • Performance tuning beyond a solid first solver

Recommended approach

Use a hybrid formulation: sample the real surface directly, seed a warp/weft lattice in surface space, solve a pin-jointed net in 3D with tangent-plane relaxation, and project each iteration back onto the source surface.

Why this approach

  • avoids triangle-mesh dependence
  • keeps the solver physically closer to a fabric net
  • supports quads as the primary visual/output cell type
  • creates a clean path to richer GIB constraints later

Why not pure UV

  • UV distortion can dominate on trimmed or complex faces
  • multi-face transition handling becomes awkward
  • the result is less like a true pinned net

Why not pure 3D from the start

  • harder to stabilize
  • projection / convergence issues are more likely
  • slower to validate on simple surfaces

Phase 1 — solver core redesign

Objective: build a direct-surface solver that produces a stable pin-jointed net.

1. Surface sampling abstraction

Add a solver-facing geometry layer that can evaluate positions, normals, and tangents directly from a FreeCAD face or face patch. This layer should also provide boundary extraction and point-in-trim tests so the solver can stay aware of the actual trimmed domain.

  • Query Surface.valueAt(u, v), Surface.normalAt(u, v), and face trim membership
  • Support analytic faces first; keep the API generic enough for later multi-face shells
  • Expose boundary curves as anchor guides for the net

2. Net data model

Replace face-centered triangulation assumptions with a graph of pin-jointed nodes connected by warp/weft bars. The primary solver output should be quads (cells), not triangle faces.

  • Node: surface point, tangent frame, boundary/anchor state
  • Bar: connection between nodes, rest length, axial stiffness, direction (warp/weft)
  • Cell: quad formed by adjacent warp/weft bars
  • Boundary: anchored, sliding, or free edge constraints

3. GIB-style solve loop

Implement the iterative equilibrium loop in the native solver. The exact math can evolve, but the first version should be deterministic and debuggable.

  1. Create a seed lattice from target spacing and starting direction.
  2. Project lattice nodes onto the source surface.
  3. Compute bar stretch, direction error, and force imbalance.
  4. Update node positions in the local tangent plane.
  5. Re-project nodes to the surface.
  6. Enforce boundary constraints and stop when the residual is below a threshold.

4. Native solver interfaces

Keep the Python-facing contract narrow so future improvements stay isolated inside the native code.

  • solve_surface(face_or_patch, parameters) -> result
  • Result includes nodes, bars, quad cells, boundary loops, strain diagnostics, convergence data
  • Keep invalid geometry reporting explicit and structured

Phase 2 — workbench integration

Objective: route CompositeShell to the new direct-surface net solver and update the visualization path.

1. Replace mesh conversion in production

  • remove production dependence on shape2Mesh() / MeshPart.meshFromShape()
  • pass the actual support surface / face patch to the adapter
  • keep tools/draper.py as reference-only during migration

2. Update adapter behavior

  • preserve get_tex_coords(), get_boundaries(), get_strains()
  • generate quads / net lines from the solver result
  • translate solver diagnostics into DrapeStatus / DrapeError

3. Update plotting / previews

  • render the source surface in one color and the drape net in another
  • show quads or net lines instead of triangle faces for the draped result
  • keep optional plot generation behind FISHNET_PLOTS=1

4. Keep command/UI stable

The existing command and object type can remain stable while the backend changes. Any later task-panel refinements should only expose solver parameters, not alter the document-object contract.

Data model

Inputs

  • support face or shell patch
  • surface parameterization / direct sampling hooks
  • seed point / seed direction
  • fabric spacing and relaxation parameters

Intermediate representation

  • node graph with tangent frames
  • bars with rest length and stiffness
  • boundary anchors / sliding edges
  • convergence metrics and residuals

Outputs

  • quad cells / net lines
  • boundary loops
  • fabric coordinates for preview
  • strain / stretch fields

Testing

Native solver tests

  • planar face
  • cylindrical patch
  • concave trimmed face
  • step / seam-like surface
  • invalid or degenerate trim / sampling case

Integration tests

  • command → object → recompute path
  • direct-surface result is accessible from CompositeShell
  • plots render quads / net lines consistently
  • failure states stay explicit
Important: keep the tests opt-in for visual output so CI can run headless, but let developers enable plots locally when debugging surface sampling or convergence.

Risks and mitigation

  • Surface projection instability: add damping, limit step size, and clamp movement to tangent planes.
  • Trim-edge degeneracy: track boundary anchors explicitly and test boundary-sensitive cases early.
  • UV distortion / singularities: keep the solver primarily surface-embedded rather than UV-only.
  • Multi-face transition complexity: defer advanced stitching until the single-face solver is stable.
  • Visualization mismatch: update plots after solver output is stable so the pictures reflect the actual new net topology.

Deliverables

  • native surface-sampling API
  • pin-jointed net data model
  • GIB-style relaxation solver
  • Python adapter for CompositeShell
  • quad/net plot updates
  • new/updated tests for plane, cylinder, concave, and seam-like surfaces

Success criteria

  • production draping no longer depends on shape2Mesh() or MeshPart.meshFromShape()
  • the solver samples the object surface directly
  • the fishnet is represented as a pin-jointed net and outputs quads as the primary cell type
  • existing workbench consumers still use the same CompositeShell API
  • tests and plots clearly show the new topology and remain easy to debug

Next action

Implement the surface-sampling abstraction first, then the net model and iterative solve loop, and only after that switch CompositeShell and the test visualizations over to the new topology.
